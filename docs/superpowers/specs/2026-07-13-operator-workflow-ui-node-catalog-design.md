# 算子工作流前端汉化与节点目录规范设计

## 1. 背景

当前系统基于 Coze Studio 前端和工作流节点体系进行了定制，已经包含三类 HTTP 封装节点与三类 MMSI 集合运算节点。本次改造以“算子工作流”为产品名称，在保留原有页面样式、组件体系和交互逻辑的前提下，完成用户可见文案汉化、品牌文案替换、节点类型 ID 重新规划，以及添加节点面板的分类调整。

本次没有需要兼容的存量工作流数据，因此允许直接修改六个自定义节点的持久化类型 ID，不设计旧 ID 迁移逻辑。

## 2. 目标

1. 首次访问默认使用简体中文，同时保留原有语言切换与偏好记忆能力。
2. 将用户可见的 `Coze`、`Coze Studio` 和 `扣子`品牌文案统一替换为“算子工作流”。
3. 将 HTTP 封装节点与 MMSI 业务逻辑节点分配到不同的 ID 段和添加面板分组。
4. 保留添加面板现有交互与绝大多数现有分类，仅隐藏“知识库&数据”。
5. 保留 Coze Studio 原有视觉体系，只做满足上述目标所需的最小化样式调整。

## 3. 非目标

- 不重构页面布局、工具栏、画布、节点卡片或配置面板交互。
- 不更换组件库、主题色、字体、图标体系或动效体系。
- 不修改内部包名、导入路径、代码变量、类型名称、埋点标识、接口字段、版权头或内部注释。
- 不为旧的 `1002` 至 `1006` 节点类型提供兼容、迁移或回退逻辑。
- 不改变六个自定义节点的表单数据结构、执行逻辑、输入输出协议或错误处理行为。

## 4. 节点类型 ID 规范

前端 `StandardNodeType`、前端 Registry 的 `type` 与 `meta.nodeDTOType`、后端 `NodeTypeMetas.ID` 必须保持一致。

| 分组 | 节点 | 新 ID | 后端分类 Key |
|---|---|---:|---|
| 算子 | 区域船舶提取 | `1001` | `operator` |
| 算子 | 低速事件筛选 | `1002` | `operator` |
| 算子 | 折返事件明细 | `1003` | `operator` |
| 算子业务逻辑 | 交集 | `2001` | `operator_logic` |
| 算子业务逻辑 | 并集 | `2002` | `operator_logic` |
| 算子业务逻辑 | 差集 | `2003` | `operator_logic` |

节点 ID 仅改变类型身份。节点目录、表单、数据转换器、`node-test.ts`、后端 adaptor 和 runner 继续沿用当前实现。

## 5. 添加节点面板

### 5.1 分类顺序

添加面板保持现有双列列表、搜索、滚动和拖拽行为，分类顺序调整为：

1. 原有无标题常用节点区域，例如工作流和注释。
2. 算子。
3. 算子业务逻辑。
4. 业务逻辑。
5. 输入&输出。
6. 组件。
7. 其他当前已启用且不属于“知识库&数据”的原有分类。

后端分类目录中，空 Key 的常用分类保持首位；`operator` 和新增的 `operator_logic` 紧随其后。前端继续消费现有节点分类数据，不另建一套面板排序状态。

### 5.2 分类内容

- “算子”只包含三个 HTTP 封装节点。
- “算子业务逻辑”只包含 MMSI 交集、MMSI 并集和 MMSI 差集。
- 原生“HTTP 请求”继续保留在“组件”分类中。
- “知识库&数据”分类从添加面板隐藏，其当前包含的“变量赋值”也不再展示。
- 其他当前已启用节点与分类继续展示。

隐藏分类只影响添加面板可见性。相关 Registry 和后端执行能力不因面板隐藏而被删除，以避免破坏工作流初始化、运行、保存和重新打开。

### 5.3 搜索行为

搜索继续复用现有节点面板实现。搜索结果只能包含面板当前允许展示的节点，因此不会通过搜索重新暴露“知识库&数据”分类中的节点。

## 6. 汉化与品牌文案

### 6.1 默认语言

- 当本地没有已保存语言偏好、用户资料也没有明确语言设置时，默认使用 `zh-CN`。
- 用户主动切换语言后，继续使用现有持久化机制保存选择。
- 不隐藏语言切换入口，不强制覆盖用户已经保存的非中文偏好。

### 6.2 汉化范围

补齐当前改造版本中实际可访问页面的用户可见英文，包括登录、工作流列表、工作流编辑器、添加节点面板、节点配置、试运行、保存发布、弹窗、空状态、Toast 和错误提示。

优先修改现有国际化资源并复用现有 `I18n` 调用。不存在国际化资源的局部文案可以接入现有国际化机制，但不得增加运行时 DOM 全局替换或字符串拦截逻辑。

### 6.3 品牌文案

- 浏览器标题、登录页、导航、空状态、帮助提示等用户可见位置中的 `Coze`、`Coze Studio` 和 `扣子`统一显示为“算子工作流”。
- 只替换品牌含义的可见文案，不替换技术类型名、代码符号、包名、路径或版权信息。
- 当文案明确指代 Coze 官方外部服务或链接时，不把真实外部服务伪装为“算子工作流”。当前系统不需要该入口时可以隐藏；必须保留时应继续显示真实服务名称。

## 7. 视觉与交互约束

- 保持原有 Coze Studio 字体、颜色、圆角、阴影、间距、按钮、弹窗和节点卡片样式。
- 不调整画布结构、工具栏位置、配置面板布局或主要操作路径。
- “算子”和“算子业务逻辑”使用现有分类标题、双列节点列表和拖拽卡片样式。
- 仅当中文或品牌替换导致文本溢出、截断或对齐错误时，进行局部宽度、间距或换行修正。
- 不增加新的主题变量、页面级皮肤或动画。

## 8. 数据流与兼容性

1. 后端节点目录接口返回分类及节点元数据。
2. 前端根据现有 enabled-node-types 白名单过滤节点，再按后端分类顺序渲染。
3. 用户拖入节点时，Registry 使用新的 `StandardNodeType` 创建节点实例。
4. 保存时 `meta.nodeDTOType` 写入新的节点类型 ID。
5. 后端通过相同的 `NodeTypeMetas.ID` 解析类型并调用既有 adaptor 与 runner。
6. 重新打开时，新的持久化 ID 匹配同一前端 Registry，恢复原节点卡片和配置表单。

由于没有存量数据，不接受同一节点同时识别新旧两个 ID，也不保留旧 ID 别名。

## 9. 错误处理

- 节点配置校验、HTTP 请求异常、MMSI 运算异常和试运行错误继续沿用现有实现。
- 分类或文案资源加载失败时继续使用当前前端降级行为，不增加新的全局错误状态。
- 节点 ID 不一致属于开发期契约错误，应通过自动化测试阻止进入运行环境，而不是在运行时猜测或转换类型。

## 10. 验证方案

### 10.1 自动化验证

- 后端测试六个新 ID 全局唯一，且节点名称、分类和 ID 映射正确。
- 前端测试 `StandardNodeType` 与六个 Registry 的 `type`、`nodeDTOType` 一致。
- 测试添加面板允许的节点集合：保留现有节点，只排除“知识库&数据”中的节点。
- 测试分类顺序：无标题常用分类、算子、算子业务逻辑，然后是其余原有分类。
- 测试首次访问默认中文、保存语言偏好后刷新仍使用用户选择。
- 对受影响包执行 TypeScript 类型检查、Lint 和相关 Vitest；对后端工作流实体与六个节点执行相关 Go 测试。

### 10.2 浏览器回归

- 确认添加面板布局、搜索、滚动、拖拽和节点卡片样式没有变化。
- 确认“知识库&数据”不显示，其他原有分类仍存在。
- 确认原有常用节点位于第一位，“算子”和“算子业务逻辑”依次位于其后。
- 分别拖入六个自定义节点，验证配置面板、单节点试运行和全流程试运行。
- 保存并重新打开工作流，确认节点类型、表单和连线完整恢复。
- 验证原生“HTTP 请求”仍位于“组件”分类并正常运行。
- 检查登录页、应用导航、工作流列表和编辑器，确认默认中文及用户可见品牌文案替换正确。
- 切换到其他语言并刷新，确认原有切换与记忆能力保持正常。

## 11. 验收标准

- 用户首次进入系统时默认看到简体中文，并能继续切换语言。
- 当前可访问页面不再以品牌文案形式显示 `Coze`、`Coze Studio` 或 `扣子`，统一使用“算子工作流”。
- 添加面板首个区域仍是原有常用节点，其后依次为“算子”和“算子业务逻辑”。
- “知识库&数据”不再显示，其他当前已启用分类不受影响。
- 六个自定义节点使用 `1001–1003` 和 `2001–2003`，前后端类型一致。
- 六个节点可添加、配置、试运行、保存并在重新打开后正确恢复。
- 页面结构和主要交互与改造前保持一致，没有主题级或布局级改动。
